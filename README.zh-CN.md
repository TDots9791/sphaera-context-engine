# Sphaera context engine

[Русский](README.md) · [English](README.en.md) · [中文](README.zh-CN.md)

面向 AI 智能体的本地、可验证上下文引擎。

Sphaera 为代码和文档建立可验证的本地工作地图。AI 只获得与当前任务相关的材料，重要结论保留回到原始来源的路径。同一张地图可服务不同智能体和模型；交付的上下文会自动适配模型可用的上下文窗口。

## Sphaera 能做什么

Sphaera 让 AI 基于项目的当前工作地图工作，而不是面对无尽的文件夹。它找到与任务相关的材料，保留其与原始来源的关联，并向每个模型交付合适规模的上下文。这减少重复阅读，加快重要结论的核验，并让团队拥有一份统一的项目记忆，而不是分散在不同 AI 工具中的对话历史。

### 项目记忆，而不是又一个 prompt

AI 不应在每次任务、更换智能体或模型时重新理解项目。Sphaera 在设备上维护持久、以证据为基础的地图：每个片段都有精确定位、来源指纹和依赖关系。文件变化时，只更新受影响的地图部分，而不是重建团队已经积累的工作认知。

模型得到的既不是“zip 压缩包”，也不是失控的长文本，而是带有必需来源的 `ContextPack`。在上下文窗口紧张时，`capsule` 将证据拆为紧凑、可逐步展开的帧：先给出路径和关键事实，再给出所需片段与关系。因此，轻量本地模型也能沿着复杂任务推进，而不会失去回到来源的路径。

### 工作体系的新能力

| 体系 | 工作方式 | 实际结果 |
|---|---|---|
| **无需模型名称目录** | Sphaera 解析供应商规则和已连接模型的可用窗口；未知 ID 不需要脆弱的例外列表 | 云端 AI 与本地 `llama.cpp` 共用一份项目上下文；已验证 6B/9B 轻量模型路径，上下文窗口可变，最高可达 2M tokens |
| **上下文交付** | `ContextPack` 和 capsule 帧保留必需证据、约束和精确 locator | 模型看到更少噪声，却能逐步展开证据，而不是根据摘要猜测 |
| **代码、文档与来源链** | 40 个结构化适配器和来源、转换、依赖结果图，将事实、推断、冲突与缺口分开 | 一份工作记忆连接代码、PDF、协议、表格、幻灯片和 AI 输出 |
| **快速完整索引** | 大规模变更时，自适应 pre-index 并行生成已验证 JSONL 分片，并通过一个 SQLite writer 提交；仅在 commit 和图验证成功后删除分片 | 更快的完整扫描，不丢失可恢复工作，也不破坏持久存储 |
| **可靠本地存储** | 压缩会创建经过验证的备份；锁与 watcher 面向独立 Sphaera 实例设计 | 多个智能体可同时维护不同项目和数据库，不发生工作体系冲突 |
| **不夸大的加速** | Sphaera 检测 CUDA 兼容 GPU；图构建、结构提取和 SQLite 仍由 CPU 执行，外部本地 `llama.cpp` worker 可使用 GPU | 加速状态保持真实：GPU 不会被包装成图加速器，但可供本地语义组件使用 |
| **集成** | 本地 CLI、MCP 以及 Codex、Claude Code、OpenCode、ZCode 的技能共用同一张地图 | 智能体在日常工作流中直接获得可验证上下文，无需单独服务器 |

### 规模：实测而非承诺

完整索引不应变成数小时等待。在包含 70 个已选文件、292 条 evidence 记录的本地文档语料验证中，staged 处理用时 **7,162 ms**，直接处理用时 **8,362 ms**；commit 后全部 70 个临时分片已删除，图仍保持有效。这是特定语料与机器上的验证，而非通用保证。方法、边界和后续测量见[性能文档](docs/zh-CN/PERFORMANCE.md)。

### AI 成本价值

- **减少重复阅读成本。** 新任务或更换模型时，无需重新加载已经熟悉的整个项目。Sphaera 复用本地工作地图，只补充已变化或当前重要的内容。
- **昂贵模型专注推理，而非搜索。** 对输入 token 成本高的模型，Sphaera 会压缩当前任务不需要的材料。实际节省可在客户自己的材料上，并结合所选 AI 供应商的用量数据核验；产品不承诺通用的投资回报百分比。
- **小模型也能获得大型任务所需的上下文。** 它们不会接收难以处理的完整档案，而是获得简明导航、关键片段、准确来源位置和连续上下文。这使经济型模型能够在自身上下文窗口内有效处理复杂项目、文档和表格。
- **Sphaera 成本可预测。** 订阅不对 AI 供应商的 token 加价；在已授权用户和设备范围内，不限制支持的 AI 智能体、模型、IDE 或平台数量。

### 无需额外基础设施的技术价值

- **不同 AI 工具共用一张地图。** Codex、Claude Code、OpenCode、ZCode 及兼容工具使用同一份本地地图。更换模型时改变的是交付的规模和形式，而不是项目记忆。
- **上下文匹配模型能力。** 小窗口获得更短、更有引导性的材料包；大窗口获得更多关联和证据。无需为每个模型单独准备数据。
- **适用于大规模材料。** 大型代码库、长篇 PDF、表格和文档链无需每次完整阅读。地图会更新受影响的数据，并引导 AI 到达相关位置，同时保留与来源的联系。
- **代码和文档处于同一工作体系。** Sphaera 支持常见编程语言，以及 DOCX、PDF、XLSX/ODS、PPTX、XML、CSV/TSV、Markdown、HTML 和 JSON。
- **俄语是第一等语言场景。** Sphaera 面向俄语文档和俄语形态设计：处理词形、西里尔字母和专业缩写，同时保留原始引文。英文材料已支持；中文材料可以建立索引，针对中文检索质量的独立 benchmark 正在准备中。
- **上下文始终关联证据。** 重要结论可追溯到代码行、页面、段落、单元格或幻灯片。来源变化后，可以识别需要复核的相关结果。

### 与同类方案的不同之处

- 许多工具擅长代码地图、相似片段检索或单个智能体记忆；Sphaera 将代码、文档、表格和结果来源整合为一个本地工作体系；
- 无需单独的服务器、图数据库或外部 Sphaera AI 模型；工作地图保留在用户设备上；
- 不必为每个 IDE 或模型建立新记忆，Sphaera 为不同 AI 工具维护一个可核验的共同基础；
- 产品区分原始材料、模型生成结果、冲突和需要复核的内容。

Sphaera 不取代专业审核，也不会使错误来源变得正确。它让从 AI 问题回到原始材料的路径更短、更可见、更便于重复利用。

### 工程深度与可验证性

质量不来自孤立命令的数量，而来自一条连贯的工作链：Sphaera 理解材料，保留其与来源的关系，按任务和模型调整交付，并使结果可复核。

| 层级 | 已验证覆盖范围 | 用户获得的价值 |
|---|---|---|
| 材料 | **40 个结构化适配器：**29 种代码模式和 11 种文档格式 | 用于代码库、表格、协议、PDF 和处理结果的统一工作体系 |
| 关系 | 一张本地的来源、转换和依赖结果地图 | 从结论回到来源，并看到变更会影响什么 |
| 上下文 | 一套数据基础，为不同 AI 模型自适应交付 | 小模型获得连贯的工作上下文；大模型不把窗口浪费在已知材料上 |
| 质量控制 | 针对已声明能力的兼容性矩阵、自动检查和 benchmark 套件 | 可依据明确版本和材料集核验声明，而不是盲目信任 |

这种组合使 Sphaera 成为一个工作体系，而非一组互不相连的功能。对客户而言，这意味着更可预测的 AI 工作、更少的重复操作，以及在财务、法律或技术决策前更快的复核。
